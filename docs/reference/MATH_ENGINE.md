# Forge Math Engine — Complete Reference

The single source of truth for every stat, every modifier mechanic, and the
pipeline that turns a `BuildCore` into `CalculatedStats`.

Generated by scanning `@wfcd/items` (1,766 mods/arcanes, 328 unique stat
strings, 7,058 parsed stat lines) and mapping each onto the engine in
`src/engine/stat-processor/` + `src/data/wfcd-resolver.ts`.

> **How to read this doc**
> * **§A** lists every stat the game exposes, grouped by domain.
> * **§B** lists every modifier mechanic (condition, stacking rule, special
>   trigger) that can attach to a stat.
> * **§C** documents the engine pipeline end-to-end.
> * **§D** is the master stat → engine-key mapping table (what works, what's
>   missing, what's wrong).
> * **§E** lists every math bug / unsupported mechanic found during the audit.

### B.4 Special Stacking Mechanics

#### B.4.1 Galvanized On-Kill Stacking
`On Kill: +X% Stat for 20s. Stacks up to Nx.`
- Base line: unconditional `+X% Stat`.
- Kill line: per-stack `+X% Stat`, multiplied by current `galvanizedStacks`
  (0…N).
- Engine: ✅ handled by `applyConditions()` with `galvanizedStacks` state.

#### B.4.2 Galvanized Crosshairs/Scope (On-Headshot, stacks)
`On Headshot: +X% Crit Chance when Aiming for 12s`
`On Headshot Kill: +X% Crit Chance when Aiming for 12s. Stacks up to 5x.`
- Two-line mods. The engine's `detectCondition` only catches the
  `when Aiming for 12s` half (via `onHeadshot` + maxStacks).
- **Bug:** the second `On Headshot Kill` line is a *different* trigger
  (`onHeadshotKill`) with its own stack cap and is currently aliased onto
  the same `onHeadshot` flag. See §E.

#### B.4.3 Blood Rush — Crit Chance × Combo Multiplier
`Critical Chance stacks with Combo Multiplier: +X%`
- Each combo tier multiplies the bonus. Real formula:
  `critChance = base × (1 + Σ critMods + bloodRushBonus × comboTier)`
- **Engine gap:** `crit_chance stacks with Combo Multiplier` is in `STAT_MAP`
  under `weapon_crit`, so it *sums* like a normal crit mod. It does **not**
  multiply by combo tier. ❌

#### B.4.4 Weeping Wounds — Status Chance × Combo Multiplier
`Status Chance per Combo Multiplier: +X%`
- Same mechanic as Blood Rush but for status. Formula:
  `statusChance = base × (1 + Σ statusMods + weepingWoundsBonus × comboTier)`
- **Engine gap:** mapped to `weapon_status` as a flat additive — does not
  scale with combo. ❌

#### B.4.5 Condition Overload / Galvanized Aptitude family
`Melee Damage per Status Type affecting the target: +X%`
`Direct Damage per Status Type affecting the target for 20s. Stacks up to 2x.`
- Final base damage multiplier:
  `1 + Σ baseDmgMods + coBonusPerStatus × activeStatusTypes`
- Engine: ✅ for the melee `co_damage` variant (line 632). ❌ for the
  Galvanized `Direct Damage per Status Type` variant — the alias in
  `STAT_MAP` sends it to the same `co_damage` key, but the `for 20s. Stacks
  up to 2x.` portion makes it an on-kill conditional. The engine's
  `detectCondition` treats `for 20s` as `galvanizedStacks`, which is
  *wrong* here — it should be `onKill` + per-status, multiplied by both
  active statuses AND stack count.

#### B.4.6 Primed Chamber / Charged Chamber
`Damage on first shot in Magazine: +X%`
- Only the first round in the mag gets the bonus. Average damage:
  `avgShotDamage × (1 + (chamberBonus / magazine))`
- **Engine gap:** `STAT_MAP` has no entry. Silently dropped. ❌

#### B.4.7 Meticulous Aim — Headshot/Bodyshot trade
`+X% Headshot Damage, -X% Bodyshot Damage`
- Two-line. Headshot multiplier only applies on headshots; bodyshot penalty
  always applies.
- **Engine gap:** `Bodyshot Damage` and `Headshot Damage` are not in
  `STAT_MAP`. Silently dropped. ❌

#### B.4.8 Maiming Strike — Slide Crit
`Critical Chance for Slide Attack: +X%`
- Adds to crit chance **only during slide attacks**. Engine has no concept
  of slide state.
- **Engine gap:** not in `STAT_MAP`. ❌

#### B.4.9 Weak Point Acuity
`Weak Point Critical Chance. Multishot cannot be modified.: +X%`
`Weak Point Damage: +X%`
- Headshot-only crit & damage. Also disables multishot mods.
- **Engine gap:** not in `STAT_MAP`. ❌

#### B.4.10 Heavy Attack Crit (x2)
`Critical Chance (x2 for Heavy Attacks): +X%`
- Normal attacks: +X% crit chance.
- Heavy attacks: +2X% crit chance.
- Engine: ✅ handled via `heavy_crit_chance` bucket at line 692-699.
  `heavyCritChance = baseCrit × (baseCritMult + heavyBonus × 2)`.
- **Bug:** the `× 2` is baked into the formula as `heavyCritBonus * 2`,
  but the WFCD value is *already* the doubled value (e.g. Sacrificial Steel
  gives `+220%` for normal / `+440%` for heavy as a single number — the
  mod text is "+220% Critical Chance (x2 for Heavy Attacks)"). The engine
  doubles it *again*. See §E.

#### B.4.11 Set Bonuses
`getModSet()` resolves the WFCD `modSet` path. `collectSetBonuses` requires
`count >= 2` and reads `info.stats[activeIdx]` — the *cumulative* bonus
string at the current piece count. The stat strings are raw WFCD prose
(`<+X% Stat>`); **they are not parsed into Modifiers**, only displayed.
- **Engine gap:** set bonus stats never affect `CalculatedStats`. ❌

#### B.4.12 Syndicate Procs
`'Blight'`, `'Entropy'`, `'Justice'`, `'Purity'`, `'Sequence'`, `'Truth'`
- On a 10s cooldown, the weapon releases an AoE radial effect that applies
  a status, restores shields/energy, or heals. Cosmetic in a calculator
  context — out of scope, but listed for completeness.

#### B.4.13 Archon Shard Flat Additives
- Azure: +225 Armor FLAT after mult.
- Crimson: +10% Strength MULTIPLIER.
- Amber: +7.5% Energy MULTIPLIER.
- Violet: +10% Crit Damage MULTIPLIER.
- Topaz: +25% Heat + +15% Shields.
- Emerald: +25% Corrosive.
- Tauforged: × 1.5.
- Engine: ✅ resolved via `resolveShard` (line 428).
- **Bug:** the shard `armor` FLAT collides with the `armor::warframe_armor`
  FLAT bucket. `shardArmorFlat = resolveFlat(abilityBuckets,
  'armor::warframe_armor')` (line 989) returns the *sum* of all FLAT
  entries in that bucket, but only Azure shards emit FLAT — so it works by
  accident. See §E.

#### B.4.14 Focus School Passives (per-school-mod stacking)
`Amp Critical Damage for each Zenurik School Mod`, `Status Damage for each
Vazarin School Mod`, `Movement Speed in Void Mode for each Naramon School
Mod`, `Amp Energy Regen Rate for each Madurai School Mod`, `bonus for each
Unairu School Mod`, `bonus for each Mod from a unique School`.
- These scale with the *count* of focus-school mods equipped. Engine has no
  concept of equipped focus-mod count. ❌

### B.5 Faction Damage & Double-Dipping
`Damage to <Faction>: +X%` applies multiplicatively to final damage:
`totalDamage × (1 + Σ factionMods)`.
For DoT procs (Slash bleed, Heat burn, Gas, Toxin, Electric), faction
applies **twice**: `DoT = baseDoT × (1 + faction)²`.
- Engine: ✅ for hit damage (`factionMult` at line 636-643).
- Engine: ✅ for DoT double-dip (`factionMult * factionMult` at lines 836,
  840, 844, 848, 852).
- **Bug:** `Roar` is encoded in `squadBuffs` as `faction_damage_all` but
  the UI (`use-build-planner.ts:586`) pushes it as a `base_damage`
  multiplier. That makes Roar double-dip on DoTs incorrectly (base_damage
  doesn't double-dip — faction does). See §E.

## C. The Engine Pipeline

```
BuildCore
  │
  ├─ collect mods/arcanes/shards/buffs per slot
  ├─ resolver.resolveMod/Arcane/Shard/Passive → Modifier[]
  │     WFCD levelStats string → parseStatLine → STAT_MAP → Modifier
  │     detectCondition() tags conditional modifiers
  │
  ├─ sortMods (by priority asc: 0 base, 1 core, 2 elem, 3 util, 4 crit, 5 aura)
  ├─ applyConditions (Galvanized stacks, on-kill/headshot/etc. toggles)
  ├─ combineElementals (slot-order single→dual pairing)
  ├─ adjustModifiers (aiming_ → normal stat when isAiming)
  ├─ bucketify → Map<stat::group, {flat, multiplier}>
  │
  ├─ Warframe path (calculateBuild):
  │     abilityBuckets → strength/duration/range/efficiency (cap 1.75)
  │     base_health/shield/armor/energy/sprint FLAT from passive
  │     health = base × (1 + Σ healthMods)
  │     armor = base × (1 + Σ armorMods) + Σ armorFLAT (shards)
  │     ehp  = health × (1 + armor/300) + shields
  │
  ├─ Weapon path (calculateSingleWeapon):
  │     1.  totalBase = baseDmgFlat × (baseDmgMult + coBonusPerStatus × activeStatuses)
  │     2.  factionMult = 1 + Σ factionMods
  │     3.  physical types: basePhys × finalBaseDmgMult × (1 + physTypeMult) × faction
  │         elemental types: totalBase × elemTypeMult × faction
  │         innate elementals: baseInnate × finalBaseDmgMult × faction
  │     4.  totalDamage = totalBase × (1 + elemMult + physMult) × faction
  │     5.  multishot = baseMs × (1 + Σ msMods)
  │     6.  critChance = base × (1 + Σ critMods + heavyBonus)
  │         critMultiplier = base × (1 + Σ critDmgMods)
  │         critFactor = 1 + critChance × (critMultiplier − 1)
  │     7.  fireRate, statusChance, reloadSpeed, magazine
  │     8.  avgShotDamage = totalDamage × multishot × critFactor
  │     9.  burstDps = avgShotDamage × fireRate
  │     10. sustainedDps = (avgShotDamage × magazine) / (mag/fireRate + reload)
  │     11. statusProbs = calculateStatusProbabilities(damagePerType)
  │         (physical × 4 weight, elemental × 1 weight)
  │     12. DoT (slash/heat/toxin/gas/electric) × viral uptime
  │     13. enemyDamage (per-type hit multiplier, armor DR, TTK, shots-to-kill)
  │
  └─ returns CalculatedStats { strength, duration, range, efficiency,
        health, shields, armor, energy, ehp, sprintSpeed, weapons{}, companion,
        setBonuses, helminthAbility, shieldRecharge, parkourVelocity,
        aimGlideDuration, enemy }
```

### C.1 Bucket Math (`bucketify`)
All modifiers sharing `stat::stackingGroup` collapse into one
`ResolvedBucket`:
```
flat       = Σ FLAT values
multiplier = 1 + Σ (MULTIPLIER values)
```
Then:
- `resolveFlat(key)` → `bucket.flat`
- `resolveMultiplied(key)` → `bucket.flat × bucket.multiplier`
- `resolveStatWithBase(flatKey, multKey)` → base × (1 + Σ mods)

## E. Identified Bugs & Math Errors

### E.1 Critical Chance × Heavy Attack Double-Doubling
`src/engine/stat-processor/:699`
```ts
const heavyCritChance = baseCrit * (baseCritMult + heavyCritBonus * 2);
```
WFCD's `Critical Chance (x2 for Heavy Attacks)` value is *already* the
doubled value — e.g. Sacrificial Steel stores `+220%` for normal and the
`(x2 for Heavy)` annotation means heavy uses `+440%`. The engine doubles the
bonus again, producing `+660%` effective heavy crit.

**Fix:** `heavyCritChance = baseCrit * (baseCritMult + heavyCritBonus)`
where `heavyCritBonus` is the raw mod value × 2 (already doubled by
WFCD). Or split the stat into two buckets.

### E.2 Blood Rush / Weeping Wounds Don't Scale With Combo
`STAT_MAP` maps:
- `Critical Chance stacks with Combo Multiplier` → `crit_chance::weapon_crit`
- `Status Chance per Combo Multiplier` → `status_chance::weapon_status`

Both are summed like normal additive crit/status mods. In-game, the bonus
multiplies by the current combo tier (1x, 1.5x, 2x, 2.5x, 3x for tiers
0–4). Without combo state in `BuildCore`, the engine can't apply this
correctly.

**Fix:** Add `comboTier` (or `comboMultiplier`) to
`ConditionalTriggerState`. Map these two stats to dedicated buckets
(`crit_chance_combo::weapon_crit`, `status_chance_combo::weapon_status`)
and apply: `bonus × comboTier` at resolve time.

### E.3 Galvanized "Direct Damage per Status Type" Mis-Conditioned
`src/data/wfcd-resolver.ts:191`
```
'Direct Damage per Status Type affecting the target for 20s. Stacks up to 2x.'
→ { stat: 'co_damage', ... }
```
`detectCondition` sees `for 20s. Stacks up to 2x.` and tags it
`galvanizedStacks`. But this modifier is *also* `per Status Type` — so the
final value should be:
`coBonusPerStatus × activeStatuses × galvanizedStacks`
The current `applyConditions` scales by stacks only, then
`calculateSingleWeapon` scales by active statuses. That *happens* to
produce the right number, but only because the two scalings are
multiplied in sequence — fragile, and there's no `onKill` gate (a Galvanized
Aptitude with 0 stacks currently contributes `0` correctly, but a Galvanized
Aptitude with stacks and 0 active statuses contributes `0` which is right
too). Acceptable, but should be explicitly modeled.

### E.4 `heavy_efficiency` / `heavy_wind_up` Read with `resolveFlat` ✅ RESOLVED
`src/engine/stat-processor/:738-739` → now reads `weaponBuckets.get(...)?.multiplier`
directly in `weapon-calc.ts:163-166`. No longer uses `resolveFlat`.

**Old code:**
```ts
const heavyWindUp = resolveFlat(weaponBuckets, 'heavy_wind_up::weapon_heavy') || 1.0;
const heavyEfficiency = Math.min(0.9, resolveFlat(weaponBuckets, 'heavy_efficiency::weapon_heavy'));
```

**Fixed code:**
```ts
const heavyWindUpBucket = weaponBuckets.get('heavy_wind_up::weapon_heavy');
const heavyWindUp = heavyWindUpBucket ? heavyWindUpBucket.multiplier : 1.0;
const heavyEfficiencyBucket = weaponBuckets.get('heavy_efficiency::weapon_heavy');
const heavyEfficiency = Math.min(0.9, Math.max(0, (heavyEfficiencyBucket?.multiplier ?? 1) - 1));
```

### E.5 `combo_chance` Read with `resolveMultiplied` ✅ RESOLVED
`src/engine/stat-processor/:742` → now reads `weaponBuckets.get('combo_chance::weapon_combo')?.multiplier`
directly in `weapon-calc.ts:169-170`. Falls back to 0 (no combo chance) when
no mods are present, which is correct.

**Old code:** `const comboChance = resolveMultiplied(weaponBuckets, 'combo_chance::weapon_combo') || 1.0;`

**Fixed code:**
```ts
const comboChanceBucket = weaponBuckets.get('combo_chance::weapon_combo');
const comboChance = comboChanceBucket ? (comboChanceBucket.multiplier - 1) : 0;
```

### E.6 `reload_speed` / `reload_speed_mult` Bucket Confusion
- `resolveWeaponPassive` emits `reload_speed::weapon_reload` FLAT (= base
  reload time in seconds).
- `STAT_MAP` maps `Reload Speed` mod → `reload_speed_mult::weapon_reload`
  MULT.
- `STAT_MAP` maps `Reload Time` mod → `reload_speed_mult::weapon_reload`
  MULT — but `Reload Time` is *not* the same as `Reload Speed`. A `+X%
  Reload Time` mod means reload takes *longer*; a `+X% Reload Speed` means
  reload is *faster*. WFCD sometimes uses `Reload Time` with a negative
  sign to mean a speed buff. The engine treats both as `reload_speed_mult`,
  so `-X% Reload Time` and `+X% Reload Speed` both become positive mults —
  which is correct *only* if WFCD's sign convention is consistent. Verify.

### E.7 Attack Speed Bucket Never Read for Melee ✅ RESOLVED
`weapon-calc.ts:117-125` now reads `attack_speed::weapon_attack_speed` and
combines it with `fire_rate::weapon_fire_rate` for melee weapons.

**Fix:** `weapon-calc.ts:120` reads `attackSpeedBucket` and
`fireRate = attackSpeedFlat * combinedAttackSpeedMult` handles melee.

### E.8 Warcry Attack Speed Never Applies to Weapons ✅ RESOLVED
Now that E.7 is fixed, Warcry's `+50% Attack Speed` pushed into
`attack_speed::weapon_attack_speed` is correctly applied to melee weapons.

### E.9 `headshot_multiplier` Wired but Never Applied ✅ RESOLVED
`weapon-calc.ts:179-181` now reads `weaponBuckets.get('headshot_multiplier::weapon_crit_damage')?.multiplier`
and computes `headshotMultiplier = 2.0 + headshotMultBonus`. The
`headshotDps` value is correctly computed with the headshot multiplier.

**Fixed code:**
```ts
const headshotMultBonus = (weaponBuckets.get('headshot_multiplier::weapon_crit_damage')?.multiplier ?? 1) - 1;
const headshotMultiplier = 2.0 + headshotMultBonus;
const headshotDps = avgShotDamage * headshotMultiplier * fireRate;
```

### E.10 Set Bonus Stats Never Affect Output ✅ RESOLVED
`index.ts:140-148` now calls `resolver.resolveSetBonusStat()` which parses
`activeBonus` strings through `parseStatLine` + `STAT_MAP` and injects the
resulting Modifiers into the bucket pool via `setBonusMods` (stacking group
is inherited from the STAT_MAP entry, e.g. `ability` for Ability Strength).

**Fixed code:**
```ts
const setBonusMods: Modifier[] = [];
for (const sb of setBonuses) {
  if (!sb.activeBonus) continue;
  const parsed = resolver.resolveSetBonusStat(
    sb.activeBonus,
    `${sb.label} Set (${sb.count}/${sb.maxPieces})`,
  );
  setBonusMods.push(...parsed);
}
```

### E.11 Aura `Aura Strength` / `Enemy Armor` / `Enemy Max Shield` Never Applied
`STAT_MAP:148-150` maps these to `aura` / `aura_debuff` buckets. The engine
never reads these buckets. Corrosive Projection (−30% enemy armor) and
Shield Disruptor (+75% shield damage) have no effect.

**Fix:** Fold `enemy_armor` and `enemy_shield` aura modifiers into
`resolveEnemy` as multipliers on `def.baseArmor` / `def.baseShields`.

### E.12 `Roar` Mis-Classified as Base Damage
`use-build-planner.ts:586-590` pushes Roar as:
```ts
{ stat: 'base_damage', value: 0.3, stackingGroup: 'weapon_base_damage' }
{ stat: 'strength',   value: 0.3, stackingGroup: 'ability' }
```
But `game-data.json:638` defines Roar as `faction_damage_all` — a
*final-damage* multiplier that should double-dip on DoTs. By pushing it
into `base_damage`, Roar only single-dips and stacks additively with
Serration instead of multiplicatively.

**Fix:** Use the `squadBuffs` data from `game-data.json` instead of
hardcoded mod definitions in the hook. Add a `faction_damage_all` bucket
that folds into `factionMult` (which already double-dips on DoTs).

### E.13 Elemental Combination Order Uses `priority` Sort, Not Slot Index
`combineElementals` collects single-element mods in array order and pairs
1+2, 3+4, etc. But `sortMods` reorders by `priority` before
`combineElementals` runs (line 624: `combineElementals(conditionedMods)`).
All elemental mods share `priority: 2`, so within the elemental group the
order is whatever the input array order was — which is **mod slot order**
from `resolveMods`. That's correct for Warframe. ✅
But: Galvanized mods conditionally drop out via `applyConditions` (line 621)
*before* combination, which shifts the array indices and can pair two
elements that weren't originally adjacent. Edge case — probably fine but
worth a test.

### E.14 `Status Duration` / `Channeling *` / `Slam/Slide/Finisher` Wired but Dead
These are all in `STAT_MAP` with valid buckets but `calculateSingleWeapon`
never reads the buckets. They're silently collected and discarded.

### E.15 `Critical Chance and Damage when Aiming` Only Half-Aliased
`STAT_ALIAS:230-235` has aliases for `Critical Chance when Aiming for 12s`
etc., but the compound `Critical Chance and Damage when Aiming` (Critical
Focus) is *not* in `STAT_MAP` or `STAT_ALIAS`. The single stat string
contains both bonuses. The parser pulls the first number and treats the
whole line as one modifier — losing the second bonus entirely.

**Fix:** Detect compound `and` stats and split into two Modifier entries.

### E.16 Many Compound `\n` Stats Not Split
WFCD joins multi-stat mods with `\n` inside a single `stats[]` string. The
resolver's `getRankValue` does `item.split('\n')` (line 277), so each
newline-separated line becomes a separate `parseStatLine` call — good.
**But** several lines themselves contain multiple `+X% Stat` clauses
without a newline (e.g. `+X% Projectile Speed and +X% Beam Range`,
`+X% Archwing Speed and Damage\n+X% Shield and Armor`). These are parsed
as one stat with name `Projectile Speed and +X% Beam Range` and only the
first number is captured. The second bonus is lost.

**Fix:** Add a splitter in `parseStatLine` that detects ` and +` /
`\n+` patterns and emits multiple parsed lines.

### E.17 `Ammo Maximum` / `Max Ammo` Read but Never Output
`STAT_MAP:99-100` maps these to `max_ammo::weapon_ammo`. Engine never
reads the bucket. Result: ammo mods are silently dropped from output.

### E.18 Vazarin Focus School Pushed as `health` FLAT 0.2
`game-data.json:625` declares Vazarin as `{ stat: 'health', value: 0.2 }`
but `use-build-planner.ts` has no branch for `buffs.focusSchool ===
'vazarin'` — Vazarin is silently dropped. Same for any focus school
without an explicit `if` branch.

**Fix:** Iterate `gameData.focusSchools` and push each passive's
`{ stat, value }` into `buffMods` with the right stacking group, instead
of hardcoding branches.

### E.19 `ability_stats.ts` Dead Code
`src/ability-stats.ts` exposes `calculateAbilityStats` but
`stat-processor.ts` does its own ability math inline (line 971-975). The
standalone module is unused and its efficiency calc (`base.efficiency × (1
+ Σ)`) disagrees with the inline calc (`1 + Σ`, assuming base = 1). Pick
one and delete the other.

### E.20 Enemy `armorStripped` Order Wrong with Heat Proc
`src/engine/stat-processor/:462`
```ts
if (enemyState.heatProc) armorStripped = 1 - (1 - armorStripped) * 0.5;
```
This folds the 50% heat-proc armor strip into `armorStripped` *before*
passing it to `calcEffectiveArmor`, which then applies `(1 − armorStripped)`
again. The result is `(1 − (1 − x) × 0.5) × …` — a single 50% multiplicative
strip, which is correct. But `calcEffectiveArmor` also applies corrosive
stacks as `1 − (1 − 0.25^n)` multiplicatively, so the order of operations
between heat, strip, and corrosive is:
```
final = scaledArmor × (1 − armorStripped_with_heat) × (1 − (1 − 0.25^n))
```
That's correct for multiplicative strips. ✅ But the comment at line 462
says "Heat proc reduces armor by 50% multiplicatively with other strips"
which is what the code does — so this is fine, just confusingly named.

### E.21 Enemy `multiTarget` Applied Last, Doesn't Affect TTK
`calculateEnemyDamage` computes `multiTargetEffectiveDps = effectiveDps ×
enemy.multiTarget` (line 581). But `shotsToKill` / `timeToKill` are
single-target. For AoE weapons, the TTK display understates by ignoring
the fact that one shot damages multiple targets. Acceptable for a
single-target lab, but the multi-target DPS number is misleading when
shown next to single-target TTK.
